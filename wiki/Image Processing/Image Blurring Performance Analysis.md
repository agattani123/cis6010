# Image Blurring Performance Analysis

## Introduction

This wiki page provides an analysis of the performance characteristics of image blurring algorithms within the context of the project. The primary focus is on understanding the runtime complexity and efficiency of different approaches to image blurring, as well as exploring techniques for optimizing performance, such as input pixel sharing.

Sources: [deprecated/hw1/docs/hw1-runtimes.png](), [deprecated/hw1/docs/hw1-shared-input-pixels.png]()

## Runtime Complexity Analysis

This section examines the runtime complexity of various image blurring algorithms, as evidenced by the provided runtime analysis chart.

![Runtime Analysis Chart](https://github.com/agattani123/cis6010/raw/main/deprecated/hw1/docs/hw1-runtimes.png)

The chart compares the runtime performance of four different image blurring algorithms:

1. **Naive**: This algorithm likely represents a straightforward, unoptimized implementation of image blurring, where each output pixel is computed independently without any performance optimizations.
2. **Separable**: This algorithm appears to be an optimized version that leverages the separability property of certain image filters, allowing the blurring operation to be performed in two separate passes (horizontal and vertical) instead of a single combined pass.
3. **Shared Input Pixels**: This algorithm seems to employ a technique where input pixels are shared or reused across overlapping regions during the blurring process, reducing redundant computations.
4. **Optimized**: This algorithm likely combines multiple optimization techniques, such as separability and input pixel sharing, to achieve the best performance.

The chart clearly shows that the "Naive" algorithm has the highest runtime complexity, growing rapidly as the image size increases. In contrast, the "Optimized" algorithm exhibits the lowest runtime complexity, remaining relatively flat even for larger image sizes.

Sources: [deprecated/hw1/docs/hw1-runtimes.png]()

## Input Pixel Sharing

One of the key optimization techniques highlighted in the provided files is input pixel sharing. This approach aims to reduce redundant computations by reusing input pixels across overlapping regions during the blurring process.

```mermaid
flowchart TD
    A[Input Image] --> B[Divide into Overlapping Regions]
    B --> C[Compute Blurred Output for Each Region]
    C --> D[Merge Overlapping Regions]
    D --> E[Final Blurred Image]
```

The diagram above illustrates the general flow of an image blurring algorithm that employs input pixel sharing. The input image is divided into overlapping regions, and the blurred output is computed for each region independently. However, instead of processing each pixel in a region from scratch, the algorithm shares or reuses input pixels from neighboring regions, reducing redundant computations.

Sources: [deprecated/hw1/docs/hw1-shared-input-pixels.png]()

### Input Pixel Sharing Example

The following diagram provides a visual representation of how input pixels are shared across overlapping regions during the blurring process:

![Input Pixel Sharing Diagram](https://github.com/agattani123/cis6010/raw/main/deprecated/hw1/docs/hw1-shared-input-pixels.png)

In this example, the image is divided into four overlapping regions (A, B, C, and D). The pixels in the overlapping areas (highlighted in green) are shared and reused across multiple regions, reducing the number of computations required to blur the entire image.
